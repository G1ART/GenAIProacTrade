최상위 제품 스펙 문서

이 문서는 구현 경로를 고정하기 위한 운영 문서입니다. 상위 제품 스펙 문서와 모순되게 해석해서는 안 되며, 구현 중 판단이 충돌할 경우 항상 상위 문서를 우선합니다.

1. 문서 목적과 위상

이 문서는 Cursor Agent가 상위 제품 스펙 문서를 실제 코드와 인프라로 변환할 때 따라야 하는 최고레벨 실행 프로토콜이다. 본 문서는 제품을 재정의하지 않으며, 구현 중 선택의 우선순위·작업 순서·검수 게이트·handoff 규칙을 고정한다.

항목

내용

문서 위상

상위 제품 스펙 문서 바로 아래의 실행 운영 문서

주 독자

Cursor Agent, 대표님, 구현 감사자

길을 잃지 않게 하고, 단계별 산출물의 완성도와 일관성을 보장

판단 우선순위

제품 진실성 > point-in-time 보존 > 운영 재현성 > 속도 > 표현적 화려함

2. 절대 원칙

• Cursor는 제품 스펙 문서의 핵심 정의를 재해석하거나 완화하지 않는다.

• 결정론적 진실 엔진과 AI 해석 계층을 혼합하지 않는다. 숫자 계산, timestamp, point-in-time join, 백테스트 계산은 반드시 deterministic code가 담당한다.

• 새로운 기능은 항상 기존 북극성에 귀속되어야 하며, 임시 편의 구현이 구조를 오염시키면 안 된다.

• 한 번에 큰 리라이트를 하지 말고, Phase 단위의 명확한 목표와 acceptance gate를 둔 작은 패치 묶음으로 진행한다.

• 모든 패치는 재현 가능해야 한다. 환경 세팅, install command, run command, smoke test, rollback 경로가 문서화되지 않으면 완료로 간주하지 않는다.

3. 작업 모드

항목

내용

Plan Mode

구현 전에 문제 정의, 파일 영향 범위, 리스크, 데이터 흐름, 산출물, 테스트 기준을 명시하는 설계 모드

Execute Mode

Plan Mode가 고정된 뒤 제한된 범위 안에서 실제 파일 수정과 실행을 수행하는 모드

Audit Mode

패치 후 구조 적합성, 회귀, 문서 최신성, 제품 철학 정합성을 재검토하는 모드

Handoff Mode

다음 세션·다음 에이전트가 즉시 이어받을 수 있도록 상태, 남은 리스크, 다음 우선순위를 정리하는 모드

4. Cursor의 기본 실행 루프

• 상위 제품 스펙 문서와 본 프로토콜 문서를 먼저 읽고, 이번 작업이 어느 Phase에 속하는지 명시한다.

• 수정 대상 파일과 신규 생성 파일을 먼저 예고한다.

• 실행 전 검수 기준을 먼저 적는다. 구현보다 기준이 먼저여야 한다.

• 작업 중간에는 큰 방향 변경을 하지 말고, 변경 필요 시 이유와 영향도를 먼저 보고한다.

• 패치 종료 시 handoff 문서를 반드시 업데이트한다.

8. 금지사항

• 데이터 vendor convenience를 이유로 point-in-time 진실성을 희생하지 않는다.

• 재무 수치 산출이나 signal score를 LLM 추론값으로 대체하지 않는다.

• 문서 업데이트를 나중으로 미루지 않는다.

• 작동하는 것처럼 보이지만 테스트가 없는 패치를 완료 처리하지 않는다.

• 문제의 본질을 덮는 우회 patch를 반복하지 않는다.

10. 종료 조건

Cursor는 기능이 조금 돌아간다는 이유만으로 완료를 선언해서는 안 된다. 문서, 실행, 검수, handoff까지 모두 닫혀야 한 Phase의 작업이 종료된다. 본 프로토콜의 목적은 단순 구현 속도가 아니라, 다음 세션에서도 방향을 잃지 않는 운영 일관성을 확보하는 데 있다.
